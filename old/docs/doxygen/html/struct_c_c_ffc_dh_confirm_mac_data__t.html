Detailed Description

Definition of MAC Data structure, containing data, known to both key agreement parties and used for key confirmation.

The data should be constructed according to NIST SP 56A rev.2 standard, sec. 5.9, 5.9.1.1 and the following requirements:
  • MacData should be concatenated according to the role, performed by the user in the Key Agreement: is he partyU or partyV and is he confirmation provider (P) or recipient (R):
  • MacDataP = messageStringP||IDP||IDR{||EphemDataP}||EphemDataR{||TextP}, where each entry is formatted as follows:
  • - actually, instead letters "P" and "R" must be set "U" or "V" according to parties roles in DH Scheme;
  • - EphemData is an EphemeralPublicKey or Nonce, contributed by the party to the Agreement;
  • - messageStringP is a 6-byte string, defined in the sec.5.9.1, 5.9.2 according to used DH Scheme;
  • - TextP is an optional bit-string about Confirmation, known to both parties;
  • - each entry is formatted as length, followed by bytes-array of data: Len||Data;
  • - length (Len) of the data is formatted as 2-bytes big endianness counter;
  • - if any explicitly defined there optionally sub-entry is omitted, then its length should be zero and data array remained empty.
- total length of MacDataP shall be not great than CC_FFCDH_MAX_SIZE_OF_MAC_DATA bytes.
Note
Confirmation is possible (effective) only if Confirmation Receiver contribute an EphemeralKey or Nonce to the Key Agreement.
Said requirements are mandatory and should be agreed by both parties as a part of Key Establishment Agreement protocol.
